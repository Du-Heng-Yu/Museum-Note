# 技术选型与开发流程

## 1. 技术选型

### 1.1 核心框架

| 技术 | 选型 | 说明 |
|------|------|------|
| 跨端框架 | Expo + React Native | 一套 TypeScript 代码，iOS + Android 双端发布 |
| 语言 | TypeScript | 类型安全，开发体验好 |
| 导航 | `@react-navigation/native` + `bottom-tabs` + `stack` | 2 Tab + 4 Stack 页面结构 |
| 数据库 | `expo-sqlite` | 本地 SQLite，离线优先 |
| 文件系统 | `expo-file-system` | 照片复制到 App 私有目录 |
| 图片选择 | `expo-image-picker` | 统一处理相册选择和相机拍摄 |
| 系统相册写入 | `expo-media-library` | 相机拍摄照片同步存入系统相册 |
| 手势 | `react-native-gesture-handler` | 时间轴双指捏合缩放 |
| 动画 | `react-native-reanimated` | 缩放动画流畅渲染 |
| 持久化设置 | `@react-native-async-storage/async-storage` | 保存用户偏好（如年份/朝代模式选择） |

### 1.2 Android 特殊配置

| 配置 | 说明 |
|------|------|
| `android:hasFragileUserData="true"` | 卸载时提示用户是否保留数据 |

### 1.3 不需要的技术

- **不需要**：后端服务、云数据库、用户登录系统、推送通知
- **MVP 暂不需要**：云备份、数据导出、国际化

---

## 2. 开发流程（与 AI 协作的分步骤操作）

### 第 1 步：生成工程骨架

**目标**：跑通项目初始化 + 导航 + 数据库建表  
**推荐模型**：`Claude Opus 4.6`  
**操作**：
- 将三份需求文档（MVP需求、数据模型、页面流程）完整提供给 AI
- 要求生成：Expo 项目初始化、`package.json` 依赖、目录结构、React Navigation 路由配置（2 Tab + 4 Stack）、SQLite 建表初始化、`dynasties.ts` 完整中国朝代数据（夏→清）
- 要求先不实现 UI，只跑通导航跳转和数据库初始化

**验收标准**：`npx expo start` 能运行，Tab 切换正常，数据库建表无报错

---

### 第 2 步：实现数据层（CRUD）

**目标**：完成两张表的增删改查函数  
**推荐模型**：`Claude Opus 4.6`  
**操作**：
- 提供数据模型文档 + 照片存储策略
- 要求生成：`db/` 目录下的数据访问函数（创建展览、查询展览列表、创建文物、查询文物、按展览查询文物、删除文物（含照片清理）、删除展览（含级联删除和照片清理）、更新展览、更新文物）
- 要求生成：照片复制工具函数（临时路径 → 私有目录）、相机拍摄保存到系统相册的函数

**验收标准**：写一个简单测试页面，能插入、查询、删除数据，照片能复制到私有目录

---

### 第 3 步：实现 FAB + Bottom Sheet + 文物编辑页

**目标**：打通录入文物的完整流程  
**推荐模型**：`Claude Opus 4.6`  
**操作**：
- 提供页面流程文档 3.3（FAB 流程）和 3.4（文物编辑页）
- 要求实现：FAB 按钮样式、Bottom Sheet 三选项、调用相册/相机、文物编辑页全部字段（含年份-朝代联动）、所属展览下拉 + 现场新建展览、保存逻辑（含照片复制）

**验收标准**：能通过三种入口成功录入一件文物，照片保存正确，年份-朝代联动正常

---

### 第 4 步：实现时间轴页

**目标**：核心功能页，纵向时间轴 + 文物卡片 + 缩放  
**推荐模型**：`GPT-5.3-Codex`（对手势和动画实现更稳）  
**操作**：
- 提供页面流程文档 3.1（时间轴页完整设计）
- 要求实现：朝代骨架渲染（真实年份比例）、文物卡片（正方形、照片填充、白字名称、占位图）、双指缩放（2-6 列动态切换、6 列时隐藏文字）、点击跳转文物详情
- 这是难度最高的页面，可能需要多轮迭代

**验收标准**：时间轴能渲染、缩放流畅、卡片显示正确、点击跳转正常

---

### 第 5 步：实现文物详情页

**目标**：查看、编辑、删除文物  
**推荐模型**：`Claude Sonnet 4.6`（UI 页面中等复杂度，Sonnet 足够且更快）  
**操作**：
- 提供页面流程文档 3.5（文物详情页）
- 要求实现：照片横向滚动、信息展示、展览名称可点击跳转、编辑跳转、删除（含照片清理和二次确认）

**验收标准**：详情页完整显示、编辑后数据更新、删除后照片清理

---

### 第 6 步：实现展览管理（列表 + 详情 + 编辑）

**目标**：展览列表页、展览详情页、展览编辑页（新建/编辑模式）  
**推荐模型**：`Claude Sonnet 4.6`  
**操作**：
- 提供页面流程文档 3.2 + 3.6 + 3.7
- 要求一次实现三个页面：展览列表（含空状态引导）、展览详情（含文物卡片列表）、展览编辑（含新建/编辑模式切换、删除二次确认 + 级联删除 + 照片清理）

**验收标准**：展览 CRUD 全流程跑通，级联删除正确

---

### 第 7 步：UI 打磨与交互优化

**目标**：视觉统一、交互细节、空状态、loading 等  
**推荐模型**：`GPT-5.3-Codex`（按原型/截图调整 UI 最擅长）  
**操作**：
- 逐页截图当前 UI，告诉 AI 哪些要调整
- 统一配色、字体、间距
- 优化空状态引导文案、保存/删除的 Toast 提示
- 键盘弹出时页面滚动适配

**验收标准**：双端视觉一致，交互无明显卡顿或布局错位

---

### 第 8 步：Android 特殊配置

**目标**：配置卸载保留数据  
**推荐模型**：`Claude Sonnet 4.6`（简单配置任务）  
**操作**：
- 在 `app.json` 或 Expo 插件中配置 `android:hasFragileUserData="true"`

**验收标准**：Android 模拟器卸载 App 时弹出"保留数据"提示

---

### 第 9 步：测试与 Bug 修复

**目标**：全流程走查，修复问题  
**推荐模型**：`Auto`（日常小改、Bug 修复用 Auto 最省额度）  
**操作**：
- 在 iOS 模拟器 + Android 模拟器上完整走一遍所有流程
- 记录问题截图，逐个让 AI 修复
- 重点测试：照片存储/删除、级联删除、时间轴缩放边界、年份-朝代联动

---

### 第 10 步：构建与上架准备

**目标**：EAS Build 打包、应用商店上架  
**推荐模型**：`Claude Opus 4.6`（上架流程涉及配置细节和审核规范）  
**操作**：
- 配置 `eas.json`，执行 `eas build`
- 准备应用商店素材：图标、截图、描述文案、隐私政策
- iOS：App Store Connect 提审
- Android：Google Play Console 提审
- 重点关注：相机/相册权限声明、隐私政策（纯本地存储，不采集用户数据）

---

## 3. AI 模型使用速查

| 场景 | 推荐模型 | 原因 |
|------|----------|------|
| 需求梳理、架构决策、上架审核 | Claude Opus 4.6 | 长链路推理、产品抽象 |
| 工程骨架、数据层、核心逻辑 | Claude Opus 4.6 | 代码生成质量高、理解复杂上下文 |
| 手势交互、动画、按原型调 UI | GPT-5.3-Codex | 前端交互实现稳定 |
| 中等复杂度 UI 页面 | Claude Sonnet 4.6 | 速度快、质量够、省额度 |
| 简单 Bug 修复、文案修改 | Auto | 最省额度 |
| 不确定时 | Claude Opus 4.6 | 兜底最稳 |
