# 数据模型文档

## 1. 数据库概述

- **数据库引擎**：SQLite（通过 `expo-sqlite` 接入）
- **存储位置**：设备本地，离线可用
- **表数量**：2 张表（`exhibitions`、`artifacts`）
- **朝代数据**：不入库，以前端常量维护（见第 3 节）

---

## 2. 数据表结构

### 2.1 exhibitions（展览表）

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | INTEGER | PRIMARY KEY AUTOINCREMENT | 主键，自增 |
| name | TEXT | NOT NULL | 展览名称 |
| museum | TEXT | NOT NULL | 博物馆名称 |
| visit_date | TEXT | NOT NULL | 参观日期，ISO 8601 格式（如 `2026-04-03`），用户填写 |
| description | TEXT | | 展览说明，可为空 |
| created_at | TEXT | NOT NULL | 记录创建时间，ISO 8601 格式，系统自动写入 |

**建表 SQL：**
```sql
CREATE TABLE IF NOT EXISTS exhibitions (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  name        TEXT    NOT NULL,
  museum      TEXT    NOT NULL,
  visit_date  TEXT    NOT NULL,
  description TEXT,
  created_at  TEXT    NOT NULL
);
```

---

### 2.2 artifacts（文物表）

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | INTEGER | PRIMARY KEY AUTOINCREMENT | 主键，自增 |
| name | TEXT | NOT NULL | 文物名称 |
| year | INTEGER | NOT NULL | 年份，负数表示 BC（如 -771 表示公元前 771 年） |
| dynasty | TEXT | NOT NULL | 中国史朝代，照说明牌原文填写（如"西周"、"战国晚期"） |
| exhibition_id | INTEGER | NOT NULL, FOREIGN KEY → exhibitions(id) | 所属展览，必填，关联展览表 |
| photos | TEXT | | App 私有目录图片路径数组，JSON 序列化存储（如 `["{documentDirectory}photos/artifact_1_0.jpg",...]`）；照片在保存时从临时目录复制到 `documentDirectory/photos/`，不引用系统相册路径；可为空 |
| description | TEXT | | 文物说明，照说明牌原文录入，可为空 |
| note | TEXT | | 个人备注，用户自由填写，可为空 |
| tags | TEXT | | 标签数组，JSON 序列化存储（如 `["青铜器","礼器"]`），可为空 |
| created_at | TEXT | NOT NULL | 记录创建时间，ISO 8601 格式，系统自动写入 |

**建表 SQL：**
```sql
CREATE TABLE IF NOT EXISTS artifacts (
  id             INTEGER PRIMARY KEY AUTOINCREMENT,
  name           TEXT    NOT NULL,
  year           INTEGER NOT NULL,
  dynasty        TEXT    NOT NULL,
  exhibition_id  INTEGER NOT NULL REFERENCES exhibitions(id),
  photos         TEXT,
  description    TEXT,
  note           TEXT,
  tags           TEXT,
  created_at     TEXT    NOT NULL
);
```

---

### 2.3 表关系说明

```
exhibitions (1) ──── (N) artifacts
```

- 一个展览包含多件文物
- 每件文物必须归属一个展览（`exhibition_id` 必填）
- 查询某展览下所有文物：`SELECT * FROM artifacts WHERE exhibition_id = ?`
- 删除展览时需同步处理关联文物（MVP 阶段建议：提示用户确认，级联删除）

## 3. 朝代常量（非数据库）

朝代数据以 TypeScript 常量维护，位于 `src/constants/dynasties.ts`，不存入数据库。

**原因**：
- App 纯离线，数据库更新与代码更新同样需要发版，无需分离
- TypeScript 常量有类型安全保障，代码可读性更高
- 时间轴渲染时直接内存访问，响应速度最快

**数据结构：**
```typescript
export interface Dynasty {
  id: number;
  name: string;       // 朝代名称，如"西周"
  startYear: number;  // 起始年份（负数=BC），如 -1046
  endYear: number;    // 结束年份，如 -771
  order: number;      // 排序序号，控制时间轴显示顺序
}

export const DYNASTIES: Dynasty[] = [
  // 示例
  { id: 1, name: "夏", startYear: -2070, endYear: -1600, order: 1 },
  { id: 2, name: "商", startYear: -1600, endYear: -1046, order: 2 },
  { id: 3, name: "西周", startYear: -1046, endYear: -771, order: 3 },
  // ...
];
```

**用途：**
- 时间轴页面渲染朝代分组骨架
- 添加文物时，根据用户输入的 `year` 自动推算并高亮对应朝代
- 用户选择朝代时，自动填入该朝代起始年份作为 `year` 参考值

## 4. 照片存储策略

**存储位置**：`expo-file-system` 的 `documentDirectory/photos/` 目录（App 私有沙盒）

**命名规则**：`artifact_{id}_{index}.jpg`（如 `artifact_3_0.jpg`、`artifact_3_1.jpg`）

**流程：**
1. 用户通过相册选择或相机拍摄，`expo-image-picker` 返回临时路径（cache 目录）
2. 用户点击"保存"时，照片从临时路径复制到 `documentDirectory/photos/`
3. 数据库 `photos` 字段存储复制后的私有目录路径
4. 相机拍摄的照片另外通过 `expo-media-library` 的 `saveToLibraryAsync()` 同步保存一份到系统相册

**删除同步**：删除文物时，同步删除 `documentDirectory/photos/` 中对应的照片文件；级联删除展览时，同步删除展览下所有文物的照片文件

**平台差异**：
- **Android**：配置 `android:hasFragileUserData="true"`，卸载 App 时系统提示用户是否保留数据
- **iOS**：卸载即清除 App 私有数据，无法保留

---

## 5. 应用层数据处理约定

| 字段 | 写入时 | 读取时 |
|------|--------|--------|
| photos | 保存时：临时路径通过 `FileSystem.copyAsync()` 复制到 `documentDirectory/photos/`，再 `JSON.stringify(string[])` | `JSON.parse(string)` |
| tags | `JSON.stringify(string[])` | `JSON.parse(string)` |
| created_at | `new Date().toISOString()` | 直接使用 |
| visit_date | 用户选择后转为 `YYYY-MM-DD` | 直接使用 |
| year | 负数表示 BC，正数表示 AD | 显示时按正负判断加 "BC"/"AD" 前缀 |
